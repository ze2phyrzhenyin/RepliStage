# 审查与优化建议

本文档记录 2026-04-06 对当前代码库的最新审查结论，基于当前真实实现，而不是旧版单剧本结构。

## 1. 当前已经完成的重要改造

### 入口流程已重构

首页现在只做模式选择：

- 开始排练
- 导演模式

剧本来源选择已经前移到 setup 页，不再把首页做成混合入口页。

### 示例剧本库已抽象出来

当前已不再把“默认剧本”写死成唯一主线，示例剧本库由 `sample-plays.ts` 管理，支持：

- `Cendrillon`
- `秦琼卖马`

### JSON 导入工作流已成型

排练模式和导演模式都可以导入 JSON，且已具备：

- 结构校验
- 业务校验
- 结构预览

### 道具布景已统一

门和椅子已经不是特殊散字段，而是统一进入 `StageProp` 模型。

### 测试基础已经补上

当前已有 5 组逻辑测试，覆盖播放器、校验、存储迁移和编辑核心。

## 2. 当前仍值得关注的问题

### 本地存储仍是唯一持久化方案

这是当前最大现实限制。

影响：

- 换设备不会同步
- 没有版本历史
- 覆盖导入后无法回滚

这不是 bug，但会限制真实长期使用。

### 示例剧本库元数据仍偏轻

`sample-plays.ts` 目前已有标题、描述和语言标签，但如果后面剧本继续增多，建议继续补：

- 封面图或主题色
- 场次数缓存
- 默认场次 id
- 角色总数
- 事件总数

### 组件级交互测试偏少

逻辑层已经有护栏，但以下层面仍缺：

- setup 流程交互
- 导演导入面板交互
- StageCanvas 高层拖拽交互

### 旧兼容字段仍存在

`StageConfig` 里还保留：

- `doorX`
- `doorY`
- `chairX`
- `chairY`

这对兼容旧数据有用，但长期应逐步收敛到 `props`。

## 3. 当前最合理的优化优先级

1. 给 setup 流程补交互测试
2. 给导演模式导入 / 导出流程补更高层测试
3. 给示例剧本库补更完整元数据
4. 评估是否加入“导入覆盖前确认 + 差异提示”
5. 长期再考虑远程存储和版本管理

## 4. 不建议现在做的事

### 不建议重新把首页做复杂

首页已经完成从“混合大厅”到“模式入口”的收敛，不应再把剧本详情、角色卡、场景说明重新堆回首页。

### 不建议把导入 JSON 提升成首页三级入口

当前正确结构是：

- 首页选模式
- setup 选来源

这比首页三入口更清晰。

### 不建议马上做大规模后端化

当前产品仍以本地单机工作流为主，先把前端流程、测试和文档打稳，收益更高。

## 5. 当前整体判断

现在这套仓库已经从“单示例 demo”进化成了“通用剧本工作台”的第一版：

- 结构已经清晰
- 工作流已经成立
- 数据模型已经统一
- 文档和测试正在补齐

接下来更应该做的是继续完善稳定性和使用体验，而不是再次推翻入口结构。
