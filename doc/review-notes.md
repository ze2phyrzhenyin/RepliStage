# 审查与优化建议

本文档记录 2026-04-06 对当前代码库的一次实现审查，目标是帮助后续维护者快速识别高优先级问题、已修复项和下一步优化方向。

## 1. 本次已修复

### 排练页编排草稿按场次隔离

此前排练页“编排”模式把自定义事件统一写入同一个 `localStorage` 键，切换到其他场次时会错误复用上一场的草稿。

现在已经改为：

- 当前键名：`stagecue_custom_events_v1:<sceneId>`
- 旧键名：`stagecue_custom_events_v1`
- 读取时兼容旧键
- 新保存会自动迁移出旧键

这能避免多场次剧本在排练端互相串台。

## 2. 当前主要风险

### 缺少自动化测试

播放器、导演模式导入、排练页编排都依赖一系列事件类型和边界条件，但仓库当前仍没有单元测试或集成测试。

最建议先补的测试对象：

- `lib/player.ts`
- `lib/play-schema.ts`
- `lib/customScript.ts`

### JSON 结构校验已经有，但业务校验还不够

当前 `zod` 校验能保证字段类型大体正确，但还没有覆盖这类业务规则：

- `actorId` 是否真的存在于当前场次
- `move_path` 是否至少有一个路径点
- `enter` / `move` 是否同时具备 `x` 与 `y`
- `line` 事件是否应该要求 `text`

这意味着“格式合法但业务不可用”的 JSON 仍可能被导入。

### 导演模式与排练编排存在重复编辑能力

现在仓库里同时有：

- 排练页里的轻量 `DirectorPanel`
- 导演模式里的完整 `EventEditor`

两者都能编辑事件、插入事件、处理路径与局部排练草稿。短期内这没问题，但长期维护会增加交互逻辑和字段支持的分叉风险。

## 3. 可以继续优化的点

### 给 PlayContext 增加版本号与迁移策略

`stagecue_current_play_v1` 目前直接存整份剧本。后续如果字段升级，没有显式 migration 机制，只能依赖运行时兜底回退。

建议：

- 为存储对象增加 `version`
- 在读取时集中做迁移
- 迁移失败时给出更清晰提示，而不是静默回退

### 减少深拷贝的重复实现

当前多处使用 `JSON.parse(JSON.stringify(...))` 做深拷贝。对这个项目目前够用，但可维护性一般，而且会在数据结构升级时掩盖不可序列化字段问题。

建议封装一个统一的 `clonePlay()` / `cloneScene()` 辅助函数，避免散落复制逻辑。

### 增加导入前后的差异提示

导演模式已经有导入预览，但还没有“覆盖了哪些内容”的差异视图。

建议增加：

- 剧目标题变化
- 场次数变化
- 每场事件数变化
- 新增或删除角色

这样会比单纯结构预览更适合真实排练编辑流程。

## 4. 文档与实现已同步的重点

本轮已修正文档中的这些过时描述：

- 不再把 `app/` 描述为包含 API Route
- 不再把导演模式写成“开发态专用”
- 不再声称排练端仍固定耦合第一场
- 明确说明排练页草稿现在按场次隔离

## 5. 推荐的后续优先级

1. 给 `lib/player.ts` 增加单元测试，先覆盖 `enter`、`move`、`line`、`exit`、`scene_end`
2. 给 JSON 导入增加跨字段业务校验
3. 给 `PlayContext` 和排练草稿增加版本迁移策略
4. 评估是否把排练编排与导演模式统一为同一套事件编辑内核
